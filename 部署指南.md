# 部署指南

适用于把项目部署到另一台 Windows 11 电脑，并尽量稳定地长期运行。

本文只保留部署、验证和长期运行真正需要的内容。

## 1. 适用场景

这份指南默认你是在一台长期在线的 Windows 11 笔记本或台式机上运行本项目，目标是：

- 能正常接入企业微信机器人长连接
- 能调用模型服务
- 能处理 PDF / Office 附件
- 能访问本地目录或共享目录
- 能较稳定地 24 小时运行

## 2. 前置要求

部署前先确认以下条件：

- Windows 11
- Node.js 18+
- Python 3
- Git
- 可访问模型服务
- 已拿到企业微信机器人的 `BOT_ID` 和 `SECRET`

建议额外确认：

- `node -v`、`npm -v`、`python --version` 可以正常执行
- 电脑已接通电源
- 网络稳定，能访问你的模型 API
- 如果要连接 NAS 或共享盘，当前机器对目标路径有读写权限

## 3. Windows 11 运行前设置

如果这台机器要长期挂着跑，先把系统层面的不稳定因素处理掉。

建议至少做以下设置：

- 电源模式设为高性能或平衡，但接通电源后不要自动睡眠
- 关闭自动休眠
- 合上盖子时设为“不采取任何操作”
- 尽量使用有线网络；如果只能用 Wi‑Fi，关闭网卡节能
- 不要把项目目录放在 OneDrive、桌面同步目录或容易被自动清理的路径下

一个推荐做法是把项目放在固定目录，例如：

```text
D:\Mycode2025\wxwork-bot
```

## 4. 拉取项目

```powershell
git clone <你的仓库地址>
cd wxwork-bot
```

如果是已有代码目录，先确认你当前使用的是预期分支和版本。

## 5. 创建 `.env`

在项目根目录创建 `.env` 文件：

```env
BOT_ID=你的企业微信机器人ID
SECRET=你的企业微信长连接密钥
OPENAI_API_KEY=你的主模型密钥
MARKITDOWN_OCR_OPENAI_API_KEY=你的OCR回退模型密钥
DASHSCOPE_API_KEY=你的阿里百炼OCR密钥
```

说明：

- `BOT_ID`、`SECRET` 用于企业微信长连接
- `OPENAI_API_KEY` 用于主模型调用
- `DASHSCOPE_API_KEY` 用于当前默认 OCR 链路
- `MARKITDOWN_OCR_OPENAI_API_KEY` 用于 OCR 回退链路

如果以后你明确不使用某条 OCR 回退链路，可以再精简环境变量；首次部署建议先保留。

## 6. 检查 `config.json`

首次部署时，至少检查下面这些配置项。

### 6.1 主模型配置

当前仓库默认主模型配置在 `agent` 下。

通常需要确认：

- `agent.model`
- `agent.openai.baseURL`

如果你打算继续使用仓库里的默认模型服务地址，可以不额外写 `MODEL_NAME` 和 `OPENAI_BASE_URL`。

如果你要切换模型服务，再按需要补充环境变量或直接修改配置。

### 6.2 合同归档目录

重点看：

- `contractMcp.libraryRoot`
- `contractMcp.dbPath`

当前仓库默认值适合本地测试。如果你要接真实共享目录：

- 建议直接写 UNC 路径，例如 `\\192.168.20.100\共享目录\合同管理`
- 不要依赖 `Z:\` 这种映射盘符
- 原因是手工启动时映射盘可能可见，但作为服务或计划任务运行时通常不可见

如果只是本地调试，可以保持默认本地目录不变。

### 6.3 企业微信通道配置

建议确认：

- `channel.type` 为 `wxwork`
- `channel.wxwork.debug` 生产环境建议改为 `false`
- `channel.wxwork.streamingResponse` 保持开启即可

说明：

- `debug=true` 方便调试，但长期运行会产生更多日志输出
- 正式部署建议关闭，除非你正在排查问题

### 6.4 OCR 与附件解析配置

重点看：

- `agent.attachmentExtraction.markitdown.enabled`
- `agent.attachmentExtraction.markitdown.timeoutMs`
- `agent.attachmentExtraction.markitdown.ocrConcurrency`

建议：

- 普通笔记本长期运行时，`ocrConcurrency` 设为 `1` 或 `2`
- 不建议一开始就把 OCR 并发开得太高
- 如果机器性能一般、模型接口波动大，宁可慢一点，也优先保证稳定

### 6.5 本地数据目录

建议确认以下路径都在本机固定目录下：

- `agent.sessionDb`
- `storage.tempDir`
- `storage.userRootDir`

这些目录会持续产生运行时数据，不要放到同步盘或临时目录。

## 7. 安装依赖

在项目根目录执行：

```powershell
npm install
```

安装过程中会自动执行 `markitdown/setup.js`，用于：

- 创建项目内 Python 虚拟环境
- 安装 `markitdown`
- 安装 `markitdown-ocr`
- 安装 Python 侧依赖

如果自动安装失败，优先检查：

- `python` 或 `py -3` 是否可用
- `pip` 是否可联网
- 当前机器是否能访问依赖源

然后手动重试：

```powershell
npm run markitdown:install
```

## 8. 首次启动

先用前台方式启动一次，确认环境没有问题：

```powershell
npm start
```

启动成功后，程序会建立企业微信长连接并开始接收消息。

如果启动阶段直接报错，优先检查：

- `.env` 是否缺变量
- 模型密钥是否正确
- Python 环境是否安装成功
- `config.json` 中的目录路径是否可访问

## 9. 首次验证

建议按下面顺序做一次完整验证。

### 9.1 文本消息验证

给机器人发一条普通文本，确认：

- 能收到消息
- 能正常返回回复
- 控制台没有持续报错

### 9.2 附件消息验证

再发一个 PDF 或 Office 附件，确认：

- 附件能成功下载
- 能正常进入解析链路
- 机器人能返回处理结果

### 9.3 测试验证

如需跑完整测试：

```powershell
npm test
```

如需单独验证 OCR 链路：

```powershell
npm run test:markitdown-ocr
```

只有文本消息正常但附件不正常时，通常优先排查 Python 依赖、OCR 密钥或附件解析配置。

## 10. 正式运行方式

不要长期靠一个手工打开的终端窗口运行。

如果目标是长期稳定运行，优先建议使用 `NSSM` 把程序托管为 Windows 服务。

### 10.1 推荐方式：NSSM

`NSSM` 适合把 `node index.js` 这种常驻进程注册成 Windows 服务。

优点：

- 开机自启
- 崩溃后自动拉起
- 不依赖登录后手工开终端
- 比任务计划程序更适合长期常驻

### 10.2 安装思路

先准备好：

- `node.exe` 的实际路径
- 项目根目录
- `index.js` 的绝对路径

例如：

- `node.exe`: `C:\Program Files\nodejs\node.exe`
- 项目目录: `D:\Mycode2025\wxwork-bot`
- 入口文件: `D:\Mycode2025\wxwork-bot\index.js`

然后安装 `NSSM`，并以管理员身份打开 PowerShell 或 CMD。

### 10.3 建议的 NSSM 配置

假设服务名叫：

```text
wxwork-bot
```

可以使用 `nssm install` 打开图形配置界面，或直接用命令写配置。

推荐配置如下。

#### Application

- `Path`
  - `C:\Program Files\nodejs\node.exe`
- `Startup directory`
  - `D:\Mycode2025\wxwork-bot`
- `Arguments`
  - `index.js`

说明：

- `Path` 必须是 `node.exe`，不是项目目录
- `Startup directory` 必须是项目根目录，否则 `.env`、`config.json`、相对路径都可能出问题
- `Arguments` 直接写 `index.js` 即可

#### Details

- `Display name`
  - `wxwork-bot`
- `Description`
  - `WxWork contract bot service`
- `Startup type`
  - `Automatic`

#### Log on

如果项目只访问本机目录，通常可以先用本地账号或 `LocalSystem` 测试。

如果项目要访问共享目录或 NAS，建议：

- 使用一个明确有共享目录权限的 Windows 账号运行服务
- 不要默认假设 `LocalSystem` 能访问网络共享

这是 Windows 服务部署里最常见的坑之一。

#### I/O

建议把标准输出和标准错误都落到日志文件：

- `Output (stdout)`
  - `D:\Mycode2025\wxwork-bot\logs\service.out.log`
- `Error (stderr)`
  - `D:\Mycode2025\wxwork-bot\logs\service.err.log`

建议提前创建：

```text
D:\Mycode2025\wxwork-bot\logs
```

#### Exit actions

建议设置：

- 第一次退出：`Restart`
- 第二次退出：`Restart`
- 后续退出：`Restart`
- `Restart delay`
  - `5000` 毫秒左右

这样即使 Node 进程异常退出，也会自动拉起。

### 10.4 命令行配置示例

下面给一个可直接参考的 `NSSM` 命令行示例。

先安装服务：

```powershell
nssm install wxwork-bot "C:\Program Files\nodejs\node.exe" index.js
```

再补充关键参数：

```powershell
nssm set wxwork-bot AppDirectory "D:\Mycode2025\wxwork-bot"
nssm set wxwork-bot DisplayName "wxwork-bot"
nssm set wxwork-bot Description "WxWork contract bot service"
nssm set wxwork-bot Start SERVICE_AUTO_START
nssm set wxwork-bot AppStdout "D:\Mycode2025\wxwork-bot\logs\service.out.log"
nssm set wxwork-bot AppStderr "D:\Mycode2025\wxwork-bot\logs\service.err.log"
nssm set wxwork-bot AppRotateFiles 1
nssm set wxwork-bot AppRotateOnline 1
nssm set wxwork-bot AppRotateSeconds 86400
nssm set wxwork-bot AppRotateBytes 10485760
nssm set wxwork-bot AppExit Default Restart
nssm set wxwork-bot AppRestartDelay 5000
```

如果需要指定服务运行账号，可继续设置：

```powershell
nssm set wxwork-bot ObjectName ".\你的Windows用户名" "你的密码"
```

如果要访问 NAS 或共享目录，通常应该使用对该共享目录有权限的账号，而不是默认系统账号。

### 10.5 启动与检查

配置完成后：

```powershell
nssm start wxwork-bot
```

检查服务状态：

```powershell
nssm status wxwork-bot
```

也可以在 Windows 服务管理器里查看是否已启动。

首次启动后建议立刻检查：

- 服务是否成功启动
- 日志文件是否正常写入
- 企业微信机器人是否能收发消息
- 如果配置了共享目录，是否能正常访问

### 10.6 常见问题

#### 服务能启动，但访问不到共享目录

优先检查：

- `contractMcp.libraryRoot` 是否使用 UNC 路径
- 服务运行账号是否对共享目录有权限
- 是否误用了映射盘符

#### 服务反复重启

优先检查：

- `.env` 是否缺失
- `Startup directory` 是否设置为项目根目录
- `node.exe` 路径是否正确
- 日志文件里是否有启动异常

#### 手工运行正常，服务运行不正常

通常是以下原因之一：

- 工作目录不对
- 服务账号权限不对
- 共享目录路径写法不对
- 环境变量或 `.env` 没有按项目根目录加载

## 11. 24 小时稳定运行建议

这部分是长期运行最容易忽略、但最影响稳定性的点。

### 11.1 只保留一个实例

同一个企业微信机器人同一时间只能保留一个有效长连接。

这意味着：

- 不要两台电脑同时跑同一个机器人
- 不要同时开手工终端和任务计划程序
- 不要重复启动多个进程

否则新连接会把旧连接顶掉，看起来就像“机器人偶尔失联”。

### 11.2 机器必须持续在线

如果这台电脑会睡眠、休眠、断网或被合盖挂起，机器人就会掉线。

所以建议：

- 全程接电源
- 不睡眠、不休眠
- 合盖不挂起
- 网络尽量稳定

### 11.3 共享目录优先用 UNC 路径

如果 `contractMcp.libraryRoot` 指向共享目录：

- 推荐写 `\\server\share\...`
- 不推荐写映射盘符

这是 Windows 长期运行场景里很常见的坑。

### 11.4 控制日志量

正式环境建议关闭 `channel.wxwork.debug`。

原因：

- 调试日志会更多
- 长时间运行更容易造成日志噪音
- 排错时再临时打开更合适

### 11.5 OCR 并发不要贪高

对 Win11 笔记本来说，附件 OCR 往往比纯文本处理更吃资源。

建议：

- 先用 `ocrConcurrency = 1`
- 确认稳定后再考虑调到 `2`
- 不建议一开始就设很高

## 12. 日常维护

项目运行一段时间后，以下目录和文件会持续增长：

- `storage/temp`
- `storage/users/*/attachments`
- `storage/users/*/data`
- `data/sessions.db`
- `data/attachment-extraction-cache.db`

建议定期做这些事：

1. 清理历史临时文件和无用附件
2. 备份 `data/` 和重要的 `storage/` 数据
3. 检查共享目录是否仍可访问
4. 检查磁盘剩余空间

如果长期不清理，最常见的问题是：

- 磁盘空间越来越小
- SQLite 文件越来越大
- 附件目录不断堆积

## 13. 常见问题

### 13.1 两台电脑能不能同时跑同一个机器人？

不能。

同一个机器人同一时间只能有一个有效长连接。

### 13.2 如果只想本地调试，不接真实共享目录，可以吗？

可以。

保持 `contractMcp.libraryRoot` 为本地默认目录即可，项目会在本地目录下创建归档目录和数据库。

### 13.3 为什么仓库里没有 `.env`、`storage`、`data`、`.tools`？

这些都是本地运行时文件，默认不会提交到仓库。

### 13.4 为什么手工运行正常，换成服务后访问不到共享目录？

大概率是因为你使用了映射盘符，或者服务运行账号没有共享目录权限。

优先检查：

- 路径是否改成了 UNC 路径
- 服务账号是否有共享目录权限

## 14. 最小部署清单

新电脑上至少完成以下事项：

1. 安装 Node.js 18+、Python 3、Git
2. 拉取仓库
3. 创建 `.env`
4. 检查 `config.json`，尤其是 `libraryRoot`
5. 执行 `npm install`
6. 执行 `npm start`
7. 验证文本和附件消息
8. 改成任务计划程序或服务方式运行
