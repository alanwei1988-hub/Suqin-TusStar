---
name: contract-nas-archive
description: 使用真实 NAS 合同目录进行归档和查询的工作流。
---

处理合同归档或查询时，遵循以下规则：

1. NAS 真实目录是全量物理文件主系统。人和 Agent 都使用同一套目录。
2. Agent 还维护一份独立的结构化归档数据库，但它不是全量档案库，只记录通过 `contract_archive` 由 Agent 正式归档过的合同。
3. 手动复制到 NAS 目录的文件，默认只会出现在 NAS 搜索结果里，不会自动出现在 Agent 归档数据库里。
4. 目录规则分两层理解：
   - 稳定规则：顶层目录和台账 sheet 的含义。
   - 动态事实：当前实际有哪些子目录。动态事实必须用工具实时查看，而不是死记。
5. 顶层目录参考：
   - `采购（启迪支出）`
   - `导师`
   - `第三方合作协议`
   - `非协议类盖章文件`
   - `孵化企业协议`
   - `其他协议`
   - `政府协议`
   - `专业服务收入协议（活动+算力+商业化）`
   - `MOU（框架协议）`
6. 查询时先判断目标来源：
   - `contract_search`：查 NAS 真实文件，适合回答“文件有没有”“在哪个目录”“手动复制进去的合同能不能找到”。
   - `contract_search_archive_records`：查 Agent 归档记录，适合回答“Agent 是否归档过”“归档 ID 是什么”“按金额/相对方/日期等字段筛选”。
   - 如果用户只说“找合同/查合同”而没说明是找文件还是找 Agent 记录，默认应同时查两边，并在答复中分别说明。
7. 查 NAS 文件时，优先查最可能的目录分支，再按最近时间、目录名、文件名关键词过滤，例如“算力”“GPU”“A800”“4090”。
8. 归档时，先列目录或查相似目录，确认目标分支后再准备归档。
9. 当用户第一次只说“归档”时，不要立刻正式归档。
   - 先调用 `contract_preview_archive`，整理并展示：拟归档目录、拟文件名、收入/支出判断、sheet 判断、不确定项，以及“将写入归档数据库的字段”中文清单。
   - `contract_preview_archive` 返回的 `pendingId` 是这次预览的服务端草稿标识。后续确认归档时，优先把这个 `pendingId` 传给 `contract_archive`，不要只靠模型记忆把整份 `contract` 再拼一次。
   - 不要再单独重复输出一段“合同基本信息”；若字段清单里已经包含合同名称、甲乙方、签署/生效日期、金额等内容，就以这份中文字段清单为准。
   - 必须把“最终将写入归档数据库的字段”逐项展示给上传人看清楚，并明确哪些字段还未填写；字段名一律用中文，不要显示 `contractName` 这类英文键名。
   - 只有上传人明确回复“确认”“按此归档”“直接归档”后，才调用 `contract_archive`。
   - 如果用户一开始就明确表示“不用确认、直接归档”，才可跳过确认轮次。
   - 如果用户在确认轮次里只补充少量修正信息，例如“就是一个月”“结束日期没问题”，调用 `contract_archive` 时应传 `pendingId`，并只在 `contract` 里覆盖这些变更字段。
10. 命名规则默认遵循现有规则文件：以时间开头，协议名称在后，最后为乙方。
   - 这里的“最后为乙方”只能当经验规则，不能机械套用。
   - 如果启迪在甲方或乙方中出现，目录名末尾应优先放“相对方/非启迪一方”，不要把启迪自己写到目录末尾。
11. 台账 Excel 目前继续人工维护，但不再是 Agent 默认归档链路的一部分。Agent 只负责：
   - 生成上传人确认消息
   - 上传人确认后执行 NAS 归档
   - 写入 Agent 归档数据库
12. 台账 sheet 选择原则：
   - `费用支出协议`：采购、租赁、装修、外包等我方支出类协议。
   - `有结算款项协议`：有明确金额或结算安排的收入类、合作类协议。
   - `无结算款项协议`：没有固定结算金额的框架协议、MOU、原则性合作协议。
13. 判断收入/支出时，一律站在“启迪”视角，不按甲方/乙方字面机械判断。
   - 启迪向外提供服务、交付资源、对外收款：按收入类处理。
   - 启迪向外采购服务、租赁资源、对外付款：按支出类处理。
   - 如果合同写“甲方接受乙方服务”，则服务提供方是乙方、接受方是甲方。
   - 例如：乙方是上海启迪创业孵化器有限公司，且合同写甲方接受乙方算力服务，这通常应归到启迪收入，不应归到启迪支出。
14. 调用 `contract_archive` 前至少要先明确：`contract`、`sourceFiles`、`archiveRelativeDir`、`operator`。
   - `contract` 对象本身必传；内部字段可以先按已识别到的部分填写，但不要把整个 `contract` 留空或省略。
   - `contract` 内部一律使用标准键名：`contractName`、`partyAName`、`partyBName`、`otherPartyName`、`signingDate`、`effectiveStartDate`、`effectiveEndDate`、`contractAmount`、`direction`、`uploadedBy` 等。
   - 不要写语义化别名，例如 `name`、`counterparty`、`ourEntity`、`signedDate`、`effectiveDate`、`amount`，运行时 schema 会直接报错。
